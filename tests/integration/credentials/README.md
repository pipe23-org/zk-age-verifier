# Test credentials

Credentials the integration tests present, plus the trust material that makes them
resolvable. Nothing here ships in the wheel; the verifier's trust store is empty until an
operator configures trust sources. The presenter loads entries by directory name and reads
only the manifest; the prover is the only reader of the raw mdoc bytes.

- `eu-av/` — EU age-verification mdoc: doctype `eu.europa.ec.av.1`, holding
  `age_over_18 = true` and a `nym` element. Issued by `scripts/generate_credentials.py`
  under test keys, mirroring the vendored entry's CBOR shapes. Carries its device private
  key, so the presenter can sign a fresh session transcript and drive a live presentation.
  The default credential.
- `mdl/` — a mobile driving licence: doctype `org.iso.18013.5.1.mDL`, holding
  `age_over_18 = true` in namespace `org.iso.18013.5.1`. Same shapes and keys as `eu-av`,
  leaf chaining to `test-anchor.pem`; only the doctype differs. Presented honestly it drives
  `claim-mismatch` — the verifier requires doctype `eu.europa.ec.av.1`.
- `eu-av-untrusted/` — `eu-av`'s shapes and claims under fresh keys, its leaf signed by a
  second CA generated in the same run and discarded. That CA's PEM is committed nowhere, so
  the leaf chains to no configured anchor and the credential drives `untrusted-issuer`.
- `eu-av-vendored/` — the upstream original `eu-av` mirrors: google/longfellow-zk
  `mdoc_tests[15]`, byte-exact, with the transcript it was captured with. No device key
  exists for it, so it cannot drive a live presentation; it checks our reading of foreign
  bytes against its captured transcript. google/longfellow-zk is Apache-2.0, the same
  licence as this repository.
- `test-anchor.pem` — the test CA that signs each trusted entry's leaf certificate; test
  configs name it as their `pem` trust source. Generated by
  `scripts/generate_credentials.py` (CA key held in memory and discarded; rerunning
  regenerates everything here).

## Entry layout

```
<name>/
  credential.cbor   # the mdoc, byte-exact from its source
  manifest.toml     # schema below
  leaf.pem          # the DS certificate presented as msoX5chain[0]
  device-key.pem    # constructed entries: the device private key
  transcript.bin    # vendored entries: the captured transcript
```

## manifest.toml schema

- `[credential]` — `doctype`; `source` (`"vendored"` or `"constructed"`).
- `[provenance]` — vendored: `upstream`, `index`, `commit`, `via`, `captured`.
  Constructed: `generator`, `template`, `generated`.
- `[issuer]` — `pk_x`, `pk_y`: the issuer's P-256 public key coordinates, hex.
- `[device]` — `key` (constructed: device private key PEM) or `transcript` (vendored:
  captured transcript), one or the other.
- `[trust]` — `status` (`"trusted"`: leaf chains to `test-anchor.pem`; `"untrusted"`:
  chains to no committed anchor); `certificate` (path of the leaf, relative to the
  manifest).
- `[claims."<namespace>"]` — one key per element; each value is the hex of the element
  value's CBOR encoding.
